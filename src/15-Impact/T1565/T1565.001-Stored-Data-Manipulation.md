## 这是什么？

**存储数据篡改**（T1565.001）是 **数据篡改**（T1565）的一个具体变体，属于 **影响** 阶段的攻击技术。



> ✏️ **打个比方**：就像偷偷改动了财务报表上的数字——攻击者篡改存储中的数据，影响业务决策或造成经济损失。

### 具体怎么理解？

修改数据库或文件中的已存储数据

攻击者使用这种技术时，通常是在影响阶段，想要达到特定的攻击目标。与父技术 T1565 相比，T1565.001 有自己独特的特点和使用场景。

### 为什么有效？

这种技术之所以有效，是因为：
1. **隐蔽性**：利用了正常系统功能或常见协议，不容易被发现
2. **技术门槛适中**：不需要特别高深的技术知识就能实施
3. **广泛适用**：可以在多种环境和系统中使用



## 真实攻击流程

```mermaid
graph TD
    A["攻击者获取数据库访问权限"] --> B["查询定位目标数据记录"]
    B --> C["使用SQL语句修改关键字段"]
    C --> D["数据库服务正常但返回错误数据"]
    D --> E["业务系统基于被篡改数据运行"]
    E --> F["用户和决策者无法察觉数据异常"]
    style C fill:#ff6b6b,stroke:#333,stroke-width:2px
```

**步骤详解：**

1. **获取数据库权限** - 攻击者通过SQL注入、窃取数据库凭据或利用数据库漏洞获得直接访问权限
2. **定位目标数据** - 查询数据库表结构，识别存储关键业务数据的表（如财务记录、用户权限、交易数据）
3. **篡改数据内容** - 使用UPDATE、INSERT等SQL语句修改数据库中的具体字段值（如修改账户余额、交易金额）
4. **保持服务正常** - 篡改操作故意不破坏数据库结构，确保应用服务正常运行而不触发异常告警
5. **隐蔽影响** - 业务系统继续使用被篡改的数据运行，用户和管理员在没有审计对比的情况下难以发现数据已被修改

### 主机层检测

**Windows事件ID**：
- 事件ID 4688：进程创建（监控可疑进程）
- 事件ID 4104：PowerShell执行（监控可疑脚本）

**Linux日志**：
- /var/log/auth.log：认证日志
- /var/log/syslog：系统日志

```bash
# 检测异常进程和命令执行
grep -i "suspicious" /var/log/auth.log
```

### 应用层检测

**用人话说：** 这条规则在检测存储数据篡改——攻击者直接修改数据库或文件中的已保存数据。典型手法包括：通过SQL注入执行UPDATE语句修改关键字段（如银行账户余额、用户权限、交易金额）、直接编辑配置文件篡改业务逻辑参数、或修改源代码植入后门（如XZ Utils供应链攻击案例）。检测的关键信号是：数据库审计日志中出现非应用层的直接SQL修改操作（特别是sa/admin账户的批量UPDATE）、关键配置文件的哈希值在非变更窗口期发生变化、或者数据完整性校验失败。这种攻击非常隐蔽——系统运行正常，没人知道数据已经被改了，直到错误的业务决策造成实际损失。

## 缓解措施

### 优先级1：关键措施

**限制攻击面**：减少暴露的服务和端口，降低被利用的风险

```bash
# 防火墙规则示例：只允许必要的出站连接
iptables -A OUTPUT -p tcp --dport 443 -j ACCEPT  # 允许HTTPS
iptables -A OUTPUT -p tcp --dport 80 -j ACCEPT   # 允许HTTP
iptables -A OUTPUT -p tcp -j DROP                 # 阻止其他TCP
```

### 优先级2：重要措施

**加强监控**：部署EDR和SIEM系统，实时监控异常行为

### 优先级3：建议措施
