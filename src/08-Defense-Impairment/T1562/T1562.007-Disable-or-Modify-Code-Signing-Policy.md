## 这是什么？

**禁用或修改代码签名策略**（T1562.007）是 **削弱防御**（T1562）的一个具体变体，属于 **防御削弱** 阶段的攻击技术。



> 🧹 **打个比方**：就像在离开前擦掉所有指纹——攻击者禁用Windows事件日志（Event Logging）来阻止安全事件的记录。

### 具体怎么理解？

允许未签名的恶意代码被执行

攻击者使用这种技术时，通常是在防御削弱阶段，想要达到特定的攻击目标。与父技术 T1562 相比，T1562.007 有自己独特的特点和使用场景。

### 为什么有效？

这种技术之所以有效，是因为：
1. **隐蔽性**：利用了正常系统功能或常见协议，不容易被发现
2. **技术门槛适中**：不需要特别高深的技术知识就能实施
3. **广泛适用**：可以在多种环境和系统中使用

### 检测思路

检测 禁用或修改代码签名策略 的关键是识别异常行为模式。以下是三个层面的检测方法：

### 网络层检测

**方法**：监控网络流量中的异常模式

```bash
# 检测异常的网络连接和数据传输

```

### 主机层检测

**Windows事件ID**：
- 事件ID 4688：进程创建（监控可疑进程）
- 事件ID 4104：PowerShell执行（监控可疑脚本）

**Linux日志**：
- /var/log/auth.log：认证日志
- /var/log/syslog：系统日志

```bash
# 检测异常进程和命令执行
grep -i "suspicious" /var/log/auth.log
```

### 应用层检测

**用人话说：** 攻击者禁用或修改Windows的代码签名策略（如AppLocker、Device Guard、WDAC）来允许未签名的恶意代码运行。他们可能通过组策略禁用AppLocker规则、在注册表中添加绕过路径来规避Device Guard签名检查。企业通过代码签名策略限制只能运行已签名的合法程序，如果这个策略被关掉，任何恶意软件都能自由执行。如果发现AppLocker规则被修改、WDAC策略被禁用、或者代码完整性检查相关的注册表被篡改，就是在拆除应用白名单这堵墙。

## 缓解措施

### 优先级1：关键措施

**限制攻击面**：减少暴露的服务和端口，降低被利用的风险

```bash
# 防火墙规则示例：只允许必要的出站连接
iptables -A OUTPUT -p tcp --dport 443 -j ACCEPT  # 允许HTTPS
iptables -A OUTPUT -p tcp --dport 80 -j ACCEPT   # 允许HTTP
iptables -A OUTPUT -p tcp -j DROP                 # 阻止其他TCP
```

### 优先级2：重要措施

**加强监控**：部署EDR和SIEM系统，实时监控异常行为

### 优先级3：建议措施

**安全意识培训**：教育员工识别相关攻击手法
