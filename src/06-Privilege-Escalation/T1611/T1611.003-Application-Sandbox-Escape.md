## 这是什么？

**应用程序沙箱逃逸**（T1611.003）是 **逃逸到宿主机**（T1611）的一个具体变体，属于 **权限提升** 阶段的攻击技术。

> 📚 **打个比方**：就像把动物园"猛兽区"的钢化玻璃展箱砸开一条细缝——狮子看起来还在玻璃后面老老实实待着，实际上它的爪子已经伸到了游客通道，随时可以完全破笼而出。

### 具体怎么理解？

从浏览器或应用程序沙箱中逃逸到操作系统

攻击者使用这种技术时，通常是在权限提升阶段，想要达到特定的攻击目标。与父技术 T1611 相比，T1611.003 有自己独特的特点和使用场景。

### 为什么有效？

这种技术之所以有效，是因为：
1. **隐蔽性**：利用了正常系统功能或常见协议，不容易被发现
2. **技术门槛适中**：不需要特别高深的技术知识就能实施
3. **广泛适用**：可以在多种环境和系统中使用



## 真实攻击流程

### 典型场景

攻击者在 **权限提升** 阶段使用 应用程序沙箱逃逸 技术，以下是典型的攻击步骤：

```mermaid
graph TD
    A["用户在沙箱内打开<br/>恶意文档/网页"] --> B["利用浏览器渲染引擎漏洞<br/>如V8 JavaScript引擎UAF"]
    B --> C["通过漏洞获取沙箱内<br/>代码执行能力"]
    C --> D["利用沙箱 IPC接口缺陷<br/>如Chrome Mojo接口漏洞"]
    D --> E["绕过沙箱进程隔离<br/>突破seccomp/AppArmor限制"]
    E --> F["在操作系统层面执行<br/>持久化恶意代码"]

    style C fill:#ff6b6b,stroke:#333,stroke-width:2px
    style E fill:#ff6b6b,stroke:#333,stroke-width:2px
```

**步骤详解：**

1. **触发漏洞入口**：诱导用户在沙箱环境中（如Chrome浏览器）打开特制的恶意文档或网页，触发渲染引擎漏洞（如V8引擎的Use-After-Free）
2. **沙箱内代码执行**：利用漏洞获取渲染进程内的任意代码执行能力，但此时仍受沙箱限制（无法访问文件系统、网络等）
3. **IPC接口攻击**：研究沙箱与宿主机之间的IPC通信接口（如Chrome的Mojo接口），寻找接口验证逻辑缺陷或类型混淆漏洞
4. **沙箱突破**：通过IPC漏洞向沙箱外的浏览器进程发送恶意消息，获取沙箱外的代码执行权限，绕过seccomp系统调用过滤
5. **系统级持久化**：利用浏览器的高权限（或配合其他漏洞）在操作系统层面写入持久化后门

**用人话说：** 应用沙箱逃逸是指攻击者从浏览器沙箱、PDF阅读器沙箱、Office沙箱等应用程序隔离环境中逃逸到操作系统。Chrome沙箱使用命名空间和seccomp限制渲染进程的权限，攻击者如果找到一个Chrome沙箱逃逸链（如CVE-2023-3079），就能从网页代码执行升级为操作系统级别的代码执行。沙箱逃逸通常与浏览器漏洞链配合使用——先用网页漏洞获得沙箱内的代码执行，再用沙箱逃逸突破隔离，最后用系统提权漏洞获得最高权限。

## 缓解措施

### 优先级1：关键措施

**限制攻击面**：减少暴露的服务和端口，降低被利用的风险

```bash
# 防火墙规则示例：只允许必要的出站连接
iptables -A OUTPUT -p tcp --dport 443 -j ACCEPT  # 允许HTTPS
iptables -A OUTPUT -p tcp --dport 80 -j ACCEPT   # 允许HTTP
iptables -A OUTPUT -p tcp -j DROP                 # 阻止其他TCP
```

### 优先级2：重要措施

**加强监控**：部署EDR和SIEM系统，实时监控异常行为

### 优先级3：建议措施

**安全意识培训**：教育员工识别相关攻击手法
