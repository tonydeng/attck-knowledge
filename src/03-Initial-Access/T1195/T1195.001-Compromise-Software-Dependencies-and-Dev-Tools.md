# 破坏软件依赖和开发工具 (T1195.001)

## 一句话通俗理解

> **在开源库、包管理器或开发工具中注入恶意代码，开发者下载使用就会中招**

## 30秒速查卡

| 维度 | 你需要知道的 |
|------|-------------|
| 这是什么？ | 攻击者在npm、PyPI等开源包里植入后门，你一装就中招 |
| 为什么危险？ | 开源依赖被成千上万个项目使用，一个被污染的包能影响整个供应链 |
| 谁需要关心？ | 开发人员、DevOps工程师、安全团队 |
| 你的第一步防御 | 使用依赖锁定文件（package-lock.json），定期审计依赖来源 |
| 如果只做一件事 | 在CI/CD流水线中添加依赖扫描工具，自动检测已知恶意包 |

## 难度等级

⭐⭐⭐ 高级 - 需要深入的技术知识和实践

## 这是什么？

**破坏软件依赖和开发工具**（T1195.001）是 **供应链妥协**（T1195）的一个具体变体，属于 **初始访问** 阶段的攻击技术。



> 📦 **打个比方**：就像在快递运输途中调包换成假货——攻击者在软件供应链的传播过程中植入恶意代码，等下游用户下载使用时自动触发。

### 具体怎么理解？

在开源库、包管理器或开发工具中注入恶意代码，开发者下载使用就会中招

攻击者使用这种技术时，通常是在初始访问阶段，想要达到特定的攻击目标。与父技术 T1195 相比，T1195.001 有自己独特的特点和使用场景。

### 为什么有效？

这种技术之所以有效，是因为：
1. **隐蔽性**：利用了正常系统功能或常见协议，不容易被发现
2. **技术门槛适中**：不需要特别高深的技术知识就能实施
3. **广泛适用**：可以在多种环境和系统中使用



## 真实攻击流程

```mermaid
graph TD
    A["选择高流量开源项目<br/>分析npm/PyPI/Maven等包管理器<br/>中广泛使用的依赖库"] --> B["注入恶意代码<br/>向开源库提交包含后门的<br/>Pull Request或新版本"]
    B --> C["等待开发者更新<br/>开发者执行npm update/pip install<br/>自动拉取被污染版本"] --> D["恶意代码在构建中执行<br/>后门在CI/CD流水线或<br/>开发环境中被触发"]
    D --> E["反向连接建立C2<br/>植入的恶意代码回连<br/>攻击者C2服务器"]
    style C fill:#ff6b6b,stroke:#333,stroke-width:2px
```

**步骤详解：**

1. **选择高流量开源项目** - 攻击者分析 npm、PyPI、RubyGems、Maven 等包管理器，挑选被大量企业项目引用的热门开源库作为目标
2. **注入恶意代码** - 通过社会工程学骗取维护者信任、利用过期域名劫持、或直接入侵维护者账户，向合法包中注入恶意代码
3. **等待开发者更新** - 开发者运行 npm update、pip install 或 CI/CD 自动构建时，无意识地拉取被污染的依赖版本
4. **恶意代码在构建中执行** - 注入的恶意代码在构建过程中被触发，可能是安装脚本（preinstall）、构建钩子或运行时初始化代码
5. **建立 C2 控制** - 恶意代码回连攻击者的 C2 服务器，下载第二阶段 Payload，在开发环境或生产服务器中建立持久化后门

### 主机层检测

**Windows事件ID**：
- 事件ID 4688：进程创建（监控可疑进程）
- 事件ID 4104：PowerShell执行（监控可疑脚本）

**Linux日志**：
- /var/log/auth.log：认证日志
- /var/log/syslog：系统日志

```bash
# 检测异常进程和命令执行
grep -i "suspicious" /var/log/auth.log
```

### 应用层检测

**用人话说：** 攻击者在npm、PyPI、RubyGems等公开包管理器里上传了带后门的版本——就像超市货架上悄悄放了有毒食品。经典案例：colors.js作者注入无限循环导致无数前端项目构建失败（2022）、SolarWinds攻击者向官方更新包中混入恶意代码（2020）。攻击者会用"typosquatting"（注册requesets冒充requests）或"dependency confusion"（上传和内部包同名的外部包）手法投毒。检测上，关注npm install或pip install后异常启动PowerShell/cmd的进程链、构建服务器上node.exe/python.exe产生意外子进程（如启动了curl或certutil下载文件）、以及构建产物哈希和官方发布不一致的情况。

## 缓解措施

### 优先级1：关键措施

**限制攻击面**：减少暴露的服务和端口，降低被利用的风险

```bash
# 防火墙规则示例：只允许必要的出站连接
iptables -A OUTPUT -p tcp --dport 443 -j ACCEPT  # 允许HTTPS
iptables -A OUTPUT -p tcp --dport 80 -j ACCEPT   # 允许HTTP
iptables -A OUTPUT -p tcp -j DROP                 # 阻止其他TCP
```

### 优先级2：重要措施

**加强监控**：部署EDR和SIEM系统，实时监控异常行为

### 优先级3：建议措施

**安全意识培训**：教育员工识别相关攻击手法
